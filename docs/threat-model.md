# 무엇을 보호하고, 무엇은 보호하지 않나요?

DELETE-DENIED는 모든 파일 작업을 막는 프로그램이 아닙니다. 현재 설계의 목표는 Codex가 지원되는 훅 경로로 위험한 삭제 명령을 보내기 전에, 사용자의 중요한 상위 폴더를 통째로 지우려는 경우를 거부하는 것입니다.

이 문서는 제품의 보호 범위와 한계를 설명합니다. DELETE-DENIED는 Codex가 `PreToolUse` 훅으로 전달하는 지원 범위 안의 셸 삭제 호출만 다룹니다.

## 어떤 경로를 보호하나요?

다음 경로를 “통째로 없애면 안 되는 상위 폴더”로 봅니다.

- 파일시스템 또는 볼륨의 루트
- 여러 사용자 계정이 들어 있는 부모 폴더와 현재 사용자의 home
- `Documents`, `Desktop`, `Downloads`
- iCloud Drive, OneDrive처럼 운영체제가 위 폴더를 다른 위치로 돌려놓은 실제 경로
- 현재 workspace와 그 위에 있는 부모 경로
- DELETE-DENIED가 설치한 사용자 정책, hook, 상태, manifest 파일과 그 부모 경로

정확한 프로젝트 폴더는 예외입니다. 예를 들어 `Documents/my-project`를 workspace로 열었다면 그 안의 소스, 빌드 결과, 임시 파일을 만들고 지우는 작업은 허용할 수 있습니다. 하지만 `Documents` 자체나 현재 workspace보다 위에 있는 부모를 선택하면 보호가 우선합니다.

## 어떤 호출을 검사하나요?

검사 프로그램은 Codex의 `PreToolUse` 훅에서 도구 이름 `Bash`로 전달되는 호출을 대상으로 합니다. 명령에 삭제가 명시되어 있으면 다음처럼 여러 표현을 읽을 수 있습니다.

- POSIX 명령: `rm`, `rmdir`, `unlink`, `find -delete`, `xargs rm`
- `rsync --delete`, `git clean`
- `bash -c`, `sh -c`, `zsh -c` 안에 들어간 제한된 명령
- PowerShell과 Windows `cmd`의 삭제 표현
- 제한된 Node.js·Python 인라인 삭제 API
- `..`, 검증된 환경 변수, `cd`, `git -C`, symlink 또는 junction이 포함된 경로

이 목록은 **검사 프로그램이 이해할 수 있는 문법**의 목록입니다. PowerShell이나 Node.js 문법을 파서가 읽는다는 사실만으로 Codex App이 그 호출을 이 훅으로 보낸다는 뜻은 아닙니다. 현재 증거로 실제 App 전달까지 확인된 범위를 넓혀 말하지 않습니다.

`^Bash$`는 명령 안의 `rm`을 찾는 필터가 아니라 Codex 도구 이름에 붙는 matcher(어떤 도구에 훅을 붙일지 정하는 필터)입니다. 따라서 삭제와 관계없는 Bash 호출도 짧은 검사를 한 번 거칩니다. 검사 프로그램은 호출이 끝나면 종료하며 폴더를 계속 훑거나 파일을 상시 감시하지 않습니다.

## 무엇은 보호하지 않나요?

다음 표면은 현재 제품의 보호 범위가 아닙니다.

- MCP 파일시스템 도구
- 화면이나 파일 관리 앱에서 직접 하는 삭제
- `apply_patch`와 Codex의 다른 파일 쓰기 표면
- 브라우저 다운로드, 별도 Terminal에서 사용자가 직접 실행한 명령
- SSH 대상, 원격 MCP, 클라우드 데이터베이스와 같은 원격 시스템
- Claude Code(현재 지원하지 않음)
- 모든 프로그램의 모든 삭제 API를 운영체제 수준에서 막는 기능

`node cleanup.js`, `python cleanup.py`, `./cleanup.sh`처럼 파일 안에 숨겨진 명령이 들어 있는 opaque script(내용을 읽지 않는 외부 스크립트)는 파일 전체를 읽어 대신 판단하지 않습니다. 직접 보이는 명령과 제한된 inline 표현만 평가합니다.

## 경로가 위험한지 어떻게 판단하나요?

단순히 문자열이 `/Users/alice`로 시작하는지만 보지 않습니다. 검사 순서는 다음과 같습니다.

1. 상대 경로를 현재 작업 폴더(`cwd`) 기준의 절대 경로로 바꿉니다.
2. `.`과 `..`를 경로 구성 요소 단위로 정리합니다.
3. 실제로 존재하는 경로는 canonical 경로(링크를 따라 실제 위치를 확인한 경로)로 확인하고, 없는 경로는 가장 가까운 기존 부모를 확인한 뒤 남은 부분을 붙입니다.
4. 원래 논리 경로와 canonical 경로를 모두 보호 목록과 비교합니다.
5. symlink, Windows junction/reparse point, 대소문자 규칙을 포함해 같은 경로인지 또는 한쪽이 다른 쪽의 부모인지 판단합니다.

보호 경로 자체, 보호 경로의 부모, 보호 경로를 가리키는 링크는 거부합니다. 보호 경로 아래의 구체적인 프로젝트 하위 경로는 허용할 수 있습니다. 현재 workspace 자체나 그 조상으로 올라가는 삭제도 거부합니다. 이 방식은 `/Users/alice2`를 `/Users/alice`로 잘못 보는 문자열 prefix 오류를 피합니다.

재귀 삭제의 대상이 확실하지 않으면 안전하다고 추정하지 않습니다. 다만 현재 코드에는 두 가지 제한적인 건너뛰기 규칙이 있습니다.

- **대상 확장 실패:** 비재귀 작업에서 대상 변수를 끝까지 풀지 못해도 인식된 wildcard가 없으면 그 대상 하나를 건너뛸 수 있습니다. 실제 파일인지 디렉터리인지 증명하지 않으며, 여러 대상 중 이런 대상은 각자 건너뛰고 해석된 다른 대상은 계속 검사합니다. 재귀 작업이거나 wildcard가 있으면 이 첫 번째 예외 밖이므로 거부합니다.
- **cwd 확장 실패:** `cd`의 목적지를 알 수 없는 경우에는 `rm` 또는 `unlink`이고, 비재귀·모호하지 않은 명령이며, 대상이 정확히 하나이고 wildcard와 대상 쉘 확장이 없을 때만 해당 작업 전체를 건너뛸 수 있습니다. 이 두 번째 예외 밖의 unknown/dynamic cwd 작업은 거부합니다.

두 예외와 별개로, 경로가 해석되고 보호 경로와 겹치지 않는 구체적인 프로젝트 하위 디렉터리 정리는 허용할 수 있습니다. 보호 경로 자체·부모·workspace 조상은 계속 거부하며, 해석되지 않은 재귀·wildcard 대상과 조건을 충족하지 못한 cwd 미확인 작업은 거부합니다.

## 어디서 한계가 생기나요?

DELETE-DENIED는 OS 커널 잠금이 아닙니다. 다음 상황에서는 보호를 보장할 수 없습니다.

- hook 밖의 프로그램이 hook·정책·파일 권한을 바꾸는 경우
- hook 검사 직후 실제 실행 전에 경로가 바뀌는 경쟁(TOCTOU, 검사와 실행 사이의 시간차를 노린 변경)
- 인코딩, command substitution, 동적 코드처럼 검사기가 의미를 알 수 없는 명령
- opaque script 내부에서 일어나는 삭제
- Codex가 PreToolUse에 전달하지 않는 실행 경로
- hook 실행 파일 누락, 비정상 종료, timeout 뒤 Codex가 명령을 계속 실행하는 경우

마지막 항목의 정확한 동작은 Codex 버전별로 공개 근거가 충분하지 않습니다. 알 수 없는 동작을 보호 성공으로 간주하지 않습니다.

## Full Access와 중지·재개는 어떻게 되나요?

Full Access 사용자를 자동으로 다른 권한 프로필로 바꾸지 않습니다. `Protected Paths: Enforced`는
Codex가 현재 훅 hash를 Trust하고 활성화한 경우에만 표시되며, 그때도 현재 작업의 권한이나 sandbox
설정을 몰래 바꾸지 않고 실제로 hook에 도착한 위험한 Bash 호출만 검사합니다. 따라서 Full Access의
모든 위험이나 hook 밖의 도구까지 없애는 기능은 아닙니다. Trust가 필요하면 `delete-denied
update --trust`를 명시적으로 실행하며, 이 명령은 app-server에서 exact identity를 확인한 경우에만
설정을 갱신합니다.

보호를 잠시 끄려면 `delete-denied suspend`를 실행합니다. 이 명령은 DELETE-DENIED 사용자 훅 항목만 제거하며 재부팅 뒤에도 유지됩니다. `delete-denied resume`은 같은 항목을 복원하고, Codex Trust가 있으면 `Protected Paths: Enforced`, 없으면 `Protected Paths: Awaiting Codex trust`를 표시합니다. 자세한 순서는 [중지·재개 안내](suspend-resume.md)를 참고하세요.

## 이 프로젝트가 말할 수 있는 것

다음 정도로만 설명합니다.

> DELETE-DENIED는 Codex의 `Bash` PreToolUse 훅에 도착한 호출 중, 중요한 로컬 부모 경로를 대상으로 하는 위험한 삭제 명령을 실행 전에 거부하도록 설계되었습니다.

다음은 주장하지 않습니다.

- 모든 AI, 모든 프로그램, 모든 삭제를 막는다고 말하지 않습니다.
- hook 밖의 파일 변경 도구, 파일 관리 앱, 원격 시스템을 보호한다고 말하지 않습니다.
- Full Access를 안전한 권한으로 바꾸거나 sandbox와 auto-review를 대신한다고 말하지 않습니다.
- parser가 이해하는 모든 문법이 Codex App에서 실제로 훅을 통과한다고 말하지 않습니다.
- 우회 불가능성이나 완전한 사고 복구를 보장한다고 말하지 않습니다.

공개 근거와 아직 답을 확인하지 못한 질문은 [조사 근거](research.md)에 정리되어 있습니다.
